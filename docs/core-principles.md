# task_pilot 核心原理说明

## 1. 这个工具是做什么的

`task_pilot.py` 用来监督一个已经存在的 `tmux` 会话。

这个会话里通常已经跑着一个交互式 CLI agent，例如 `codex`。`task_pilot.py` 不负责启动它，而是在外面加一层观察和推进逻辑。

它要解决的问题很简单：

- agent 可能中途停住
- agent 可能只回了一句进度说明
- agent 可能还没做完，但没有继续往下执行

`task_pilot.py` 的作用就是在这种“看起来停住了”的时候，判断要不要再推它一下。

## 2. 核心思路

它的工作方式可以概括成 4 步：

1. 定期读取目标 `tmux` 会话的输出快照
2. 判断快照有没有变化
3. 如果长时间没变化，就认为会话进入 idle
4. 对 idle 状态做一次决策：继续推进，或者继续等待

所以它本质上是一个“面向 tmux 会话的空闲监督器”。

## 3. 为什么它能工作

因为对交互式 CLI 来说，终端输出本身就能反映当前状态。

如果一个 agent 还在持续工作，`tmux` pane 里的内容通常会继续变化。  
如果 pane 内容长时间完全不变，就可以把这视为一种“可能停住了”的信号。

`task_pilot.py` 不尝试理解整个程序内部状态，而是用一种更轻量的办法：

- 抓取终端输出
- 对输出做归一化
- 计算哈希
- 比较当前哈希和上一次哈希是否相同

只要哈希不变持续超过阈值，就进入 idle 判断。

## 4. 它怎么判断 idle

程序会按固定间隔轮询目标会话。

每次轮询时，它会：

- 从 `tmux capture-pane` 抓取最近一段历史输出
- 去掉一些尾部空白差异
- 对归一化后的文本计算哈希

如果新快照和旧快照不同，说明会话还在变化：

- 更新“最后变化时间”
- 继续等待下一轮轮询

如果快照一直不变，并且持续时间超过 `idle_threshold`，就认为当前会话已经 idle。

也就是说，它不是根据 CPU、进程状态或网络状态判断 idle，而是根据“终端输出是否还在变化”来判断。

## 5. idle 之后会做什么

进入 idle 后，程序不会立刻无脑发送“继续”。

它会先进入一个决策步骤。当前实现支持两种模式：

- `rule`
- `codex`

## 6. `rule` 模式的原理

`rule` 模式最直接。

只要会话 idle 达到阈值，就发送一条固定提示词回到被监督会话里，例如：

`Please continue the current task...`

所以 `rule` 模式本质上就是：

- 检测空闲
- 到点就补一句固定的“继续做”

优点是简单、稳定、没有额外依赖。  
缺点是它并不真正理解上下文，任务已经完成时也可能继续推动。

## 7. `codex` 模式的原理

`codex` 模式多了一层分析。

当会话 idle 时，`task_pilot.py` 会把当前终端快照交给 `codex exec --json`，让它判断当前到底应该：

- `continue`
- `wait`

如果分析结果是 `continue`，程序就发送一条新的推进提示。  
如果分析结果是 `wait`，程序就什么都不发，等终端内容变化之后再允许下一次分析。

所以 `codex` 模式的本质是：

- 先检测空闲
- 再让一个外部分析器判断“这是真的卡住了，还是其实已经结束了”

这样比固定规则更保守，也更贴近上下文。

## 8. 为什么要有 `wait`

因为“终端不再变化”不一定代表任务没做完。

也可能是：

- agent 已经完成任务
- agent 已经回答完，正在等用户下一句
- 当前状态本来就应该停在这里

如果每次 idle 都强行补一句“继续”，很容易把已经完成的会话重新推乱。

所以 `codex` 模式里专门保留了 `wait` 这个结果，用来表示：

- 当前虽然 idle
- 但不应该继续推动

## 9. 为什么要限制继续次数

程序有 `max_continue` 限制。

原因是如果一个会话始终没有真正推进，监督器可能反复发送继续提示，形成无意义循环。

所以它会记录当前已经自动推进了多少轮；达到上限后就停止。

这个设计的目的不是追求“永不停止”，而是避免进入死循环。

## 10. 为什么要等快照变化后再重新分析

在 `codex` 模式下，如果分析结果是 `wait`，程序会进入“等待快照变化”状态。

这样做是为了避免同一份静止画面被重复分析很多次。

也就是说，当前实现不是：

- idle 一次就分析一次

而更接近：

- 对同一份快照最多分析一次
- 只有快照真的变了，才值得重新判断

这能减少重复分析和无意义日志。

## 11. 为什么还要做一次快照复核

`codex` 分析需要时间。

在分析进行的这段时间里，被监督会话本身可能已经继续输出了新内容。如果还按照旧快照的结论去执行，就可能发出过时的指令。

所以当前实现会在分析完成后再抓一次快照：

- 如果快照已经变了，说明结论过时，直接丢弃这次决策
- 如果快照没变，再执行决策

这一步的核心目的就是避免“基于旧画面做出新动作”。

## 12. 为什么要通过 `tmux send-keys`

因为这个工具的定位不是直接控制 agent 进程，而是监督一个已经存在的终端会话。

所以它和被监督对象交互的方式也很简单：

- 读取时用 `tmux capture-pane`
- 写回时用 `tmux send-keys`

这意味着它不依赖某个具体 agent 的内部 API，只要目标运行在 `tmux` 里，就可以套这一层监督逻辑。

## 13. 为什么要有 `start` / `stop`

程序启动后不会立刻开始自动监督，而是先等待运行时命令。

这样设计是为了把“进程已启动”和“开始自动推动”分开：

- 你可以先把监督器挂起来
- 确认目标会话正确后再输入 `start`
- 需要时再用 `stop` 暂停自动监督

它更像一个带手动开关的守护进程，而不是一启动就立即干预。

## 14. 会话状态为什么要落盘

在 `codex` 模式下，分析器会有自己的 session id。

程序把这个状态按被监督的 `tmux` 会话名存到 `.task_pilot_sessions/` 里，是为了让后续分析可以延续之前的上下文，而不是每次都从零开始。

这让分析器更像“持续跟踪这个会话”，而不是一次性判断器。

## 15. 当前设计边界

这个工具的边界很明确：

- 一次进程只监督一个 `tmux` 会话
- 它只观察终端输出，不理解 agent 内部真实状态
- 它不创建会话，也不启动 CLI
- 它只负责“是否继续推动”的监督，不负责任务内容本身

所以它不是调度平台，也不是通用 agent 框架，而是一个很轻的监督层。

## 16. 一句话总结

`task_pilot.py` 的核心原理就是：

通过轮询 `tmux` 会话输出判断 agent 是否长时间无变化；一旦进入 idle，就用固定规则或 `codex` 分析来决定是继续推动，还是继续等待。
